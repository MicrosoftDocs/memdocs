# Passwordless solutions with Microsoft Intune

Microsoft's passwordless authentication strategy helps organizations reduce reliance on passwords—one of the most common attack vectors—by enabling secure, user-friendly sign-in experiences across platforms. This article outlines how to implement passwordless sign-in using Microsoft Intune and Microsoft Entra ID, with guidance for IT administrators managing enterprise environments.

## How Microsoft's passwordless solution works

Microsoft's passwordless solution integrates Entra ID for identity and single sign-on (SSO) with Intune for device configuration and policy enforcement. This combination enables users to authenticate using strong credentials—such as biometrics, FIDO2 security keys, or passkeys—without entering passwords.

## Intune-Supported Passwordless Authentication Methods 

Intune provides first-class support for several passwordless authentication methods in the enterprise. By configuring these methods on managed devices, Intune helps replace passwords with stronger, phish-resistant credentials: 

Windows Hello for Business (WHfB) - This is Microsoft's passwordless sign-in method for Windows 10/11, using a PIN or biometrics (fingerprint/face) backed by a hardware key (TPM). Intune can enable and require Windows Hello for Business during Windows enrollment. In Intune's admin center, under Windows enrollment > Windows Hello for Business, an administrator sets "Configure Windows Hello for Business" to Enabled to turn it on tenant-wide1. Intune also lets admins enforce specific Hello settings (PIN length/complexity, allow/disallow biometrics, etc.) through policies (either the Hello for Business blade or an Identity Protection profile). When a Windows device enrolls in Intune and is Azure AD-joined, Intune can prompt the user to set up a Hello PIN/biometric during the out-of-box experience. Thereafter, the user signs into Windows with the PIN or biometric gesture, not a password. The Hello credential is tied to the device's TPM and user, and never leaves the device, making it extremely difficult to compromise. Notably, Hello for Business on an Intune-managed device can operate in key trust or certificate trust modes; with cloud trust (a recent option), no on-prem AD certificates are needed for integration. Bottom line: Intune + Hello for Business replaces Windows passwords with PINs/biometrics that are backed by asymmetric keys, greatly enhancing security. 

FIDO2 Security Keys - Intune allows organizations to use external FIDO2-compliant security keys (USB, NFC, etc.) for signing into Windows and other services. In the same Windows Hello for Business configuration, there's an option "Use security keys for sign-in", which Intune admins set to Enabled2. This policy causes Azure AD-joined Windows 10/11 machines to accept FIDO2 security keys at the logon screen. For instance, an employee could use a YubiKey or similar token: at the Windows lock screen, they plug in the key and tap it (and/or enter the key's PIN) to sign in, rather than typing a password. Intune also supports deploying FIDO2 via an Identity protection profile (which can target specific groups if a tenant-wide setting isn't desired)3. Once enabled, users must have previously registered a FIDO2 key with their Entra ID account (which is an Entra ID configuration - enabling the FIDO2 authentication method for the tenant and having users register keys through MySecurityInfo). Importantly, FIDO2 keys are portable credentials: a user can carry their key and use it to log into any Intune-managed Azure AD-joined device (if allowed), which is great for scenarios like hot-desking or shared PCs. After logging in with a FIDO2 key, the experience is the same as Hello - Entra ID issues a token and the user has SSO. From Intune's perspective, enabling FIDO2 login is a one-time config; usage is then controlled by Entra ID's policies (e.g., which users can use keys). Administrators often pair this with disabling password sign-in to truly go passwordless for key users. 

Temporary Access Pass (TAP) - A Temporary Access Pass is a time-limited, single-use passcode that Entra ID can generate as a strong credential alternative. It's often used to onboard new users or recover accounts without a password. While TAP is defined and managed in Entra ID's Authentication Methods policy, Intune plays a role on Windows devices by enabling Web Sign-In. Web Sign-In is a Windows 10/11 feature that lets the login screen accept a cloud-provided credential (like TAP or Microsoft Authenticator) via a web flow4. Intune can turn this on by a device configuration (Settings Catalog > Authentication > "Enable Web Sign-in" = Enabled). With Web Sign-in enabled, a user at the Windows login prompt can choose "Sign-in options" and select the web sign-in icon, which then prompts for a code. This is where they can enter a Temporary Access Pass provided to them. For example, a new employee who has never set a password might be issued a TAP out-of-band; at first login on an Intune-enrolled laptop, they use the TAP through web sign-in and gain entry to Windows5 6. Because TAP counts as a strong auth, the user can then immediately be prompted to set up Windows Hello for Business (completing their passwordless onboarding)7. Intune's role is crucial: without enabling Web Sign-in, the Windows UI wouldn't accept TAP. Additionally, as of the latest Windows 11 updates, Web Sign-in can also allow Authenticator app sign-in (showing a QR code to approve with your phone) as a login method8 9. This means Intune-enabled Web Sign-in provides a path for users to log in with a phone-based credential if they have no password or forgot it - further reducing dependency on passwords. TAPs are typically short-lived (for example, 1 day, one-time use), so they are a secure way to bootstrap a more permanent passwordless method. 

Passkeys - Passkey is the industry term for the next generation of passwordless credentials that can synchronize across devices (built on FIDO2 standards). In Microsoft's ecosystem, passkeys can refer to two categories: device-bound passkeys (which are essentially the same as a FIDO2 security key or Hello credential tied to one device) and syncable passkeys (which can roam via cloud backup). Entra ID has added support for passkeys as an authentication method, especially starting in 202410 11. For instance, the Microsoft Authenticator app (version 6.7+ on iOS 17/Android 14 or later) can act as a FIDO2 authenticator and store a device-bound passkey for a user's Entra account12 13. When enabled, a user can register a passkey through Authenticator (the app generates a key pair and the public key is registered in Entra ID)14. This effectively turns their phone into a login key for Entra ID accounts - similar to a hardware security key but using the phone's secure enclave and biometrics. Intune ensures that the device prerequisites are in place: for example, Intune compliance policies can enforce a minimum OS version (iOS 17/Android 14 required for passkeys support in Authenticator)15, and require the Authenticator app to be installed. There's not a "passkey policy" per se in Intune; rather, Intune's device management makes it possible to use passkeys by keeping devices up to date and managed (and marking them as compliant so that Conditional Access allows their sign-in). On Windows, a syncable passkey might be stored in Windows Hello or via a browser's password manager. Microsoft announced that Windows 11 will support issuing and using cloud-synced passkeys by late 202416, meaning a user could create a passkey on Windows and have it available on their other devices (or vice versa). In the context of Intune, once Windows supports that, admins might encourage users to use passkeys (for instance, by integrating with Edge or encouraging passwordless strategy) but again the enforcement is via Entra ID policies (e.g., disabling password fallback). Key point: Passkeys are becoming a universal credential across Windows, Mac, iOS, Android. Microsoft Entra ID's adoption of passkeys (now generally available since late 2024) means users can sign in with platform biometrics (like Face ID or Windows Hello) even for first-time sign-in on a new device by leveraging another device. Intune's job is to facilitate this by making sure devices and apps (Authenticator, Company Portal, etc.) are in place so that users can register and use passkeys easily. Many organizations are currently piloting passkeys for a truly passwordless future, and Intune provides the management layer (device trust, compliance, config) to support that transition.

## Microsoft Entra ID's Role in SSO and Passwordless 

Microsoft Entra ID is the cloud identity provider that underpins both the single sign-on experiences and the passwordless authentication methods. Once Intune has set up a device for passwordless sign-in, it's Entra ID that actually authenticates the user and gives them access tokens for services.  Some key aspects of Entra ID's role: 

Passwordless Credential Verification: Entra ID holds the public keys or secrets associated with each passwordless method in a user's account. For example, when a user sets up Windows Hello for Business, Entra ID stores a public key for that user's Hello credential. When the user signs in with Hello (say using their fingerprint), the Windows device presents a cryptographic proof (signed with the user's private key from the TPM) to Entra ID. Entra ID validates this proof against the stored public key, confirming the user's identity with high assurance27 28. Similarly, if a user registers a FIDO2 security key or Authenticator passkey with Entra ID, the service knows that for any login attempt with that method, it should expect a challenge response from that key. In short, Entra ID replaces password checking with crypto challenge-response checking. This is far more phishing-resistant - an attacker can't just steal a PIN or key stroke; they would need the physical device or biometric. 

Token Issuance and Single Sign-On: After successful authentication, Entra ID issues tokens - most importantly a Primary Refresh Token (PRT) for the device/user. On Windows Azure AD-joined machines, the PRT is stored by the OS and is automatically used to grant the user SSO to Azure AD-integrated apps (Office apps, Teams, SharePoint, etc.). On mobile devices (iOS/Android), when the user signs in via the Authenticator app or Company Portal, Entra ID similarly provides a PRT or refresh token to the app, which then brokeres it to other apps so they don't prompt again. On macOS with the Enterprise SSO plug-in, Entra ID issues a device token (in the form of a certificate or token in the user's keychain) when the user signs in with Entra ID at the macOS login window29 30. In all cases, Entra ID's token is the reason SSO works: once you've logged in, Entra ID recognizes the device and user, and any app that trusts Entra ID can leverage the existing session. For example, a user who unlocks their Intune-managed Windows laptop with Hello will get instant access to Outlook and Teams (they open without asking for credentials) because those apps detect the Entra ID PRT in Windows and use it to authenticate silently. Similarly, on an Intune-managed iPhone, after a user signs in to one Microsoft app, other Microsoft apps will also sign in silently using the token stored by Authenticator as broker. This is true single sign-on — log in once, use many apps. 

Conditional Access and Device Trust: Entra ID doesn't just blindly issue tokens; it checks conditions via Conditional Access (CA). Intune and Entra ID are integrated such that Entra ID knows the management and compliance state of the device at auth time. For instance, an Intune-managed device can be marked "compliant" (meeting all security requirements set by IT). Conditional Access policies in Entra ID can then require that a device be Intune-compliant and/or Azure AD-joined in order to allow login tokens for certain apps. If a user tries to use a passwordless method from a device that Intune reports as non-compliant or unknown, Entra ID can deny the token or require additional proof (like MFA). In practical terms, this means even a passwordless sign-in is evaluated for risk and policy: logging in with a FIDO2 key on a fully managed, compliant device might give immediate access, but the same key used on an unmanaged device might be blocked. Device registration is key here. Windows, Mac, iOS, and Android all perform an Azure AD device registration when enrolled in Intune (commonly called Workplace Join for the latter two). This registers a device object in Entra ID and establishes a trust relationship. As a result, when the user authenticates, Entra ID can bind the auth to the device and look up its status. A Primary Refresh Token issued to a device is actually scoped to that device; if the device is later found non-compliant or is disabled by IT, Entra ID can refuse to honor that PRT for new resource access. Thus, Entra ID acts as the policy brain, enforcing that passwordless sign-ins occur only from healthy, trusted devices per the organization's rules (with Intune feeding those health/trust signals). 

Cross-Platform Integration: Entra ID provides some platform-specific hooks to enhance SSO. For example, on Apple platforms, Microsoft worked with Apple to create the Enterprise SSO plug-in, which uses Entra ID endpoints. When a user signs into a Mac via Platform SSO or an iOS app via the SSO extension, the plug-in communicates with Entra ID to obtain tokens that are stored in the Apple credentials ecosystem (Keychain)31 32. On Windows, Entra ID leverages the OS capability to store tokens and certificates. On Android, Entra ID's capability to deliver a PRT is reliant on the device being registered (which Intune handles) and the presence of the Authenticator/Company Portal acting as an agent. The deep integration means each OS has a slightly different way of caching the Entra ID token, but Entra ID unifies the outcome: a token that grants access. Also, features like Entra ID Cloud Kerberos Trust illustrate integration: Entra ID can issue a Kerberos TGT for on-prem AD to a Windows device that just did a Hello login, bridging Azure AD and Active Directory33. This is Entra ID taking the passwordless cred (Hello key) and translating it to an AD ticket so the user can seamlessly access file shares or legacy apps - again, one authentication unlocking many doors. 

Phishing Resistance and Security Monitoring: Entra ID as the identity provider adds intelligent security on top of passwordless methods. Passwordless methods like Hello, FIDO, passkeys are phishing-resistant by design (they won't work if a user is tricked to a fake website - the key is bound to the real domain34). Entra ID further enhances this by detecting anomalous attempts: for example, if someone somehow tried to replay a Hello authentication from another device, Entra ID would see that the device ID or key doesn't match. Entra ID's sign-in logs will show that the authentication method used was FIDO2, Hello, etc., giving administrators insight into usage35. Additionally, Entra ID Identity Protection can flag risky sign-ins, even for passwordless (e.g., if a typically compliant device suddenly reports as jailbroken, or a normally biometric login is replaced by a PIN under unusual circumstances). In short, Entra ID ensures that going passwordless doesn't mean losing visibility or control - quite the opposite, the security signals are stronger (key-based auth and device compliance), and Entra ID can act if something seems off. 

In summary, Entra ID is the central hub that validates the user's passwordless credentials and then seamlessly signs the user into everything they need. Intune prepares the device and user to use those credentials, and Entra ID makes sure those credentials open the right doors (and only for the right people on the right devices). The tight integration between Intune and Entra ID means the identity system is always aware of device state, and the device is configured to use the identity system to its fullest - together enabling a highly secure yet frictionless login experience across all platforms. 

## Integration Points Between Intune, Entra ID, and Various Platforms 

The end-to-end passwordless SSO solution involves close cooperation between Intune (device management), Entra ID (identity), and the client operating systems (Windows, macOS, iOS, Android). Here's how these pieces come together on each platform: 

# [:::image type="icon" source="../../media/icons/platforms/windows.svg"::: **Windows**](#tab/windows)

On Windows, Intune and Entra ID integration is deepest, providing a very smooth experience: 

Azure AD Join + Intune Enrollment: Typically, a Windows device for a cloud-first organization will be Azure AD-joined (meaning it's registered directly with Entra ID as its primary domain). This often happens through Windows Autopilot or during the first sign-in, where the user enters their Entra credentials and the device joins the tenant. Auto-enrollment is usually enabled, so the moment the device Azure AD joins, it also enrolls into Intune management. This bootstrap process is the first integration point: the device now has an identity in Entra ID and is known to Intune as a managed device. Intune can evaluate compliance immediately and start applying policies (including those that enforce passwordless config like Hello). From the user perspective, this might just feel like "logging into my new PC with my work account" - behind the scenes, that action set up device trust with Entra and gave Intune control. 

Windows Hello for Business setup: With Intune policies requiring Windows Hello for Business, the user will be prompted to set up a PIN (and optionally biometric) right after Azure AD join. For example, after a user's first sign-in or during Autopilot, a "Create a PIN" dialog appears - this is coming from the Intune policy that told the device to require Hello for Business. Once the PIN and/or fingerprint is configured, the user can use it for subsequent logons. At the next login, the user sees the option to sign in with their PIN or biometric with a message like "Sign in using your PIN" instead of password. This illustrates Intune's policy integration with the Windows OS credential providers. Intune also can make Hello the default. In Windows 11 22H2 and above, Intune can deploy the "Passwordless Experience" policy which hides the password field on the login screen for users who have Hello set up36 37. If that is enabled, the user basically is not even shown a password option - they'll see only PIN or biometric prompt by default, reinforcing using Hello. (They could still click "Other user" if they absolutely needed password for some reason, but it's tucked away38.) The OS, Intune, and Entra ID are working together here to funnel the user toward the safer credential. 

Logging in and SSO: When the user unlocks Windows with Hello or a FIDO2 key, Windows uses Entra ID to validate and log them in. After the local desktop unlocks, a Primary Refresh Token is obtained from Entra ID silently. This PRT is stored by Windows (in memory, tied to the user login session). Now if the user opens any Microsoft 365 app or visits a SaaS application that federates with Entra ID, the Windows OS supplies a single sign-on token so they don't have to sign in again. For example, launching Word or Outlook automatically signs in the user with their Entra account. Even accessing third-party applications that use Entra ID (e.g., an SAP system integrated with Entra ID) via a web browser will not prompt for a password - the browser gets an Entra ID SSO token via Windows Hello (often using integrated Windows authentication or a web redirect to login.microsoftonline.com that gets answered by the PRT). The user essentially experiences true SSO across apps. If the organization has on-premises resources (files shares, older apps) and they have configured Azure AD Kerberos Trust, the user's Entra ID login can even transparently grant a Kerberos ticket so they can access those on-prem resources without an AD password39. Intune plays a supporting role by ensuring the device meets the requirements for that (e.g., a certain patch level, the right version of Windows 10/11, etc., which would be part of compliance policies). 

Policy Enforcement & Conditional Access: On Windows, Intune can enforce many security settings (disk encryption/BitLocker, antivirus, firewall, etc.) and mark the device compliant. Entra ID's Conditional Access might require a device to be Compliant and/or Azure AD Joined to let a user access sensitive apps. Because the device is Intune-managed, it continuously reports compliance. If, say, the device falls out of compliance (maybe BitLocker gets turned off), Intune will flag it, and Entra ID would start denying access tokens for certain apps until the issue is resolved. From the integration perspective, the passwordless sign-in will still succeed in unlocking the device (e.g., user can log in with Hello), but access to cloud apps can be instantly curtailed by Entra ID if the device isn't meeting standards. This dynamic is central to Zero Trust: the passwordless credential proves who the user is, and Intune's device state proves what the device's security posture is; together Entra ID decides if that combination is allowed. Intune can also push Config Profiles that directly enable some Entra ID-related features - for example, Intune can deploy a policy for "Enable Cloud Kerberos Trust" for Hello for Business if hybrid support is needed40, or deploy certificate connectors if certificate trust is the chosen Hello mode. Thus Intune not only hardens the device but also configures how it talks to Entra ID (and AD) in some cases. 

Updates and Maintenance: Intune manages Windows updates and keeps devices on supported versions. This is subtly important for passwordless: new features like the Passwordless login experience or improved FIDO2 support often come in newer Windows releases (e.g., the ability to use phone sign-in at the Windows logon came with a 2023 update41 42). By using Intune's update ring policies, IT can ensure all Windows 11 devices get those updates, thereby enabling the latest passwordless enhancements. Without Intune or similar management, some devices might not be updated and thus not benefit from these integrations (or even be missing critical security fixes). So Intune's lifecycle management feeds into the overall reliability and security of the passwordless scenario on Windows. 

In essence, on Windows devices Intune and Entra ID create a very tight-knit system: the device is joined to Entra, Intune enforces configuration (like Hello) that directly uses Entra ID, and Entra ID in turn trusts the device per Intune's reports and gives out tokens for everything. Users get a fast login (PIN or fingerprint) and then can work without signing in again, even to legacy resources, and admins get strong security (no password to steal, plus device compliance gating access). 

# [:::image type="icon" source="../../media/icons/platforms/macos.svg"::: **macOS**](#tab/macos)

Historically, Macs have been somewhat outside the traditional Windows AD/Azure AD domain join world - users log in with local accounts, and SSO to enterprise resources was limited. That has changed significantly with the introduction of the Microsoft Enterprise SSO plug-in for Apple devices and Apple's own Platform SSO framework for macOS. Intune and Entra ID now integrate to allow a near Windows-like experience on Macs: 

Microsoft Enterprise SSO Plug-in & Platform SSO: Microsoft developed an SSO plug-in that ties into Apple's enterprise authentication framework. On macOS 13+ (Ventura) and especially macOS 14 (Sonoma), Apple introduced Platform SSO, which goes a step further by allowing a cloud identity (like Entra ID) to be used at the macOS login window itself43 44. Intune can deploy a configuration for the Microsoft Enterprise SSO plug-in in Platform SSO mode, which effectively means the user will sign into the Mac using their Entra ID credentials (instead of a local username/password)45 46. This Intune policy is configured via the Settings Catalog under "Extensible Single Sign On (SSO)" settings, where the admin specifies Microsoft's plug-in bundle ID and the authentication method to use (Secure Enclave, Password, or Smartcard)47 48. When Intune pushes this policy to a Mac, a few things happen: 

The Microsoft Company Portal app (or Authenticator, which includes the plug-in) is enabled as an SSO extension on the device. 

If Platform SSO is configured, macOS will show a new login option for the user's Entra ID account. For example, instead of the user clicking on a local account to log in, they might see an "Azure AD Account" login prompt. The user enters their Entra ID email address. The first time, they'll go through a credential prompt (which could be password, or if Secure Enclave method is chosen, possibly a more guided setup to use Touch ID as a key). 

The Mac then registers with Entra ID: behind the scenes, a workplace join (WPJ) certificate is obtained and stored in the Mac's keychain49 50. The device shows up in Entra ID as Azure AD registered (or joined in a workplace context). This is similar to how an iPhone or a Windows device registers, but now it's triggered at OS login. 

Passwordless options on Mac (Secure Enclave): If the admin chose Secure Enclave Key as the authentication method in the Intune policy, the Mac will create a hardware-bound key pair in the Secure Enclave (the secure hardware module in T2 or Apple Silicon Macs) for the user's Entra ID login51 52. The user will likely set up a PIN or use Touch ID during the first Entra ID login to protect this credential (similar concept as Windows Hello, but on Mac). After that, the user can sign into their Mac with Touch ID or the PIN associated with the Secure Enclave key - essentially a passwordless login to the Mac itself using Entra ID. Importantly, this does not replace the user's local account password with their Entra password; instead, it maps the Entra ID authentication to a local account session. This is why it's passwordless and phishing-resistant - the credentials used at login are the Entra ID key pair, not a reusable password53 54. If the admin chose the "Password" method instead in Intune, then it would literally use Entra ID username/password at login (which syncs the local keychain password) - but many organizations will prefer Secure Enclave or smartcard for true passwordless. 

Seamless SSO in Apps: Once the user is logged into macOS via the Entra ID plug-in (whether by password or the passwordless Secure Enclave key), the Microsoft Enterprise SSO plug-in also takes care of application SSO. The Intune policy includes the necessary information (like redirection URLs and tenant ID) so that any app or browser that is directed to Entra ID for sign-in will use the existing session. For example, when the user launches Microsoft Teams or Outlook on the Mac, those apps call Apple's SSO extension APIs. The Microsoft plug-in sees the request and automatically provides tokens or prompts to silently sign in the user using the fact that the device is already authenticated with Entra ID55 56. As a result, the user might not see any additional login prompts after the initial OS login. This SSO even extends to Safari or Chrome - if the user goes to a SSO-protected website, the plug-in can inject the authentication token (with user consent) so they don't log in again57 58. Essentially, the Mac behaves as if it is "AD joined" but to Entra ID: sign in to device unlocks access to apps and sites. 

Device Compliance and Conditional Access: The Mac, having registered with Entra ID, now has a device object and is known to Intune. Intune can mark it as compliant if it meets requirements (like requiring FileVault disk encryption, certain OS version, no prohibited apps, etc.). Conditional Access rules can then apply to Mac access just like Windows. For instance, an admin can require that Mac devices be compliant and either Entra ID joined or registered to access Office 365. With the new Platform SSO, when the user logs in with Entra ID on the Mac, that login attempt itself can be gated by Conditional Access - e.g., perhaps you require MFA if outside network. The integration allows the Mac login to trigger an MFA prompt on the user's phone if needed, and only let them in once satisfied. After login, whenever the user gets tokens for apps, Entra ID knows the device ID and compliance state (thanks to Intune) and can allow or deny resource access accordingly. This is a major shift: previously, without the plug-in, a Mac was just a random device hitting the cloud and often fell under "unmanaged device" in Conditional Access, but now it can be treated as a known, managed endpoint with a registered device identity. 

User Provisioning and Management: Platform SSO on Mac also offers some advanced integration like Just-In-Time account provisioning - Intune's policy can specify that if the user logging in doesn't have a local account, macOS should create one for them on the fly, and even designate if they're an admin or standard user based on directory group membership59 60. For example, you might map Entra ID roles to macOS admin rights (perhaps members of an "IT Admins" Entra group get a local admin account). This mapping info is part of the Intune configuration. That means IT can manage Mac account settings centrally - something that usually required heavy AD binding or scripts in the past. Additionally, Intune can configure whether a Mac allows "Authenticated Restart" (login with network credentials when no user is cached) or even a Shared iPad-like mode called "Authenticated Guest" on Mac where multiple users can use a Mac with their own Entra IDs one at a time without persisting data61 62. These are new capabilities courtesy of macOS 14 and Intune's integration with it. 

Updates and App Deployment: Intune also manages macOS updates and can deploy the Company Portal and Office apps. Keeping macOS updated is important as Apple continues to improve the SSO framework. For instance, macOS 14 added more robust Platform SSO features that weren't in macOS 13. Intune ensures devices move to those versions. Intune can push the Microsoft Authenticator app if needed (on macOS, Company Portal contains the SSO plug-in, but Microsoft also made the Authenticator app available on macOS for just the SSO function in some cases). By deploying and maintaining these components, Intune completes the integration loop - without Intune, each Mac would have to be manually set up for the plug-in, which doesn't scale. 

With these integrations, a Mac user in an Intune/Entra ID environment can have an experience very close to a Windows user: They turn on their Mac, use Touch ID (for example) to log in with their company Entra ID account, and then they can open Outlook, Teams, SharePoint in Safari, etc., all without repeated sign-ins. And the organization can enforce compliance (for instance, require the Mac to have encryption and no risky software - otherwise Conditional Access will block data sync). This is a huge improvement in both user experience and security for Mac in the enterprise. It's worth noting these capabilities are relatively new (publicly released in 2023-2025 timeframe)63 64, so they represent the cutting edge of Intune and Entra ID integration.

# [:::image type="icon" source="../../media/icons/platforms/ios-ipados.svg"::: **iOS/iPadOS**](#tab/ios-ipados)

On iPhones and iPads, the concept of "device login" with Entra ID is not present in the same way (users unlock their device with a personal PIN/biometric, not a work account). However, Intune and Entra ID still enable a robust SSO and passwordless experience within apps on iOS: 

Device Enrollment and Broker Apps: Intune enrolls iOS devices via a management profile. As part of enrollment, typically the user is prompted to install the Company Portal app (for device management) and often the Microsoft Authenticator app (for authentication). The Authenticator app on iOS serves as the broker for authentication - it's the equivalent of having a secure token store on the device that other apps can use. When a user signs into any Microsoft app (say Outlook) on an Intune-managed iPhone, the app will either find an existing token or redirect the user to Authenticator to sign in. Because the device is Intune-registered (workplace joined to Entra ID) and Authenticator is present, upon first sign-in, Entra ID can issue a Primary Refresh Token to Authenticator for that device/user pair. This PRT is then used to silently sign in the user to other apps. Intune's contribution here is that it made sure the device is registered in Entra ID (during enrollment, a device record with a certificate is created) and that Authenticator is there to accept the token. Without Intune enrollment, the user can still use Authenticator for MFA or so, but they might not get device-wide SSO because the device wouldn't be recognized as compliant or be able to get a PRT. 

SSO App Extension (Microsoft Enterprise SSO plug-in): Apple provides an SSO app extension framework on iOS as well (since iOS 13+). Microsoft's plug-in for Apple (the same one used on Mac) also works on iOS/iPadOS to allow SSO between applications and Safari. Intune can deploy a configuration profile of type Single Sign-on extension for iOS, specifying the Microsoft Entra ID SSO extension65 66. This profile contains settings like the tenant ID, a list of allowed URLs, etc. Once applied, this means: 

If a user is in Safari or in any managed app that uses modern auth, the Microsoft SSO extension can intercept authentication requests to Entra ID and inject credentials. 

For example, suppose a user opens Safari and goes to SharePoint Online. Normally, they'd have to enter their username and maybe get a Authenticator prompt. With the SSO extension active and the user already signed in via Authenticator on the device, the extension will automatically use the existing token to log them in (or at least significantly streamline it)67 68. 

This works across Microsoft apps and any other app that uses the iOS SSO extension and is configured for Entra ID. Many Microsoft apps (like Outlook, Teams, OneDrive) natively take advantage of the broker (Authenticator) so they might not strictly need the SSO extension profile, but having it can even help those apps or line-of-business apps to share credentials. 

Authenticator App - Passwordless Sign-in: On iOS, users can set up the Microsoft Authenticator app for phone sign-in for their Entra account. Phone sign-in replaces the password: when logging into a new device or app, instead of entering a password, the user gets a prompt on Authenticator to approve (with number matching). This is a form of passwordless authentication enabled by Entra ID. Intune doesn't configure this directly, but by encouraging Authenticator use and maybe requiring MFA, many users will opt into phone sign-in. Additionally, as discussed, Authenticator on iOS 17+ can now manage Entra ID passkeys69 70. A user can create a passkey in Authenticator (which uses the iPhone's secure enclave and Face ID/Touch ID). This is stored by Authenticator (and optional iCloud Keychain backup for recovery if allowed)71 72. With that in place, when the user needs to sign into a web app on their PC, for instance, they can use that phone passkey (scanning a QR code as described in Entra ID's process)73 74. So iOS essentially becomes a provider of a credential and a mediator for SSO. Intune's job is mainly to ensure the Authenticator app is present and not tampered with (Intune can use App Protection Policies to protect Authenticator since it's an app that holds creds but on iOS it's mostly system-protected already). 

Conditional Access and Compliant Device: iOS devices in Intune have a compliance status. Often, CA policies might require "device to be marked as compliant" for certain apps on mobile. Because the device is enrolled, when the user signs in via Authenticator the first time, that sign-in will show up as coming from an "IOS" device with a device ID. Entra ID checks compliance (with a bit of a delay potentially if the device just enrolled). If compliant, the user proceeds; if not (say the user hasn't set a device PIN and the compliance policy requires one), the access can be blocked. Intune then guides the user to fix the compliance (Company Portal might notify them to set a passcode or enable encryption, etc.). In effect, to use passwordless auth on an iPhone for work, the phone itself must be secured to corporate standards - which is exactly the objective. Intune and Entra enforce that together. 

Managed Apple IDs vs Entra ID: It's worth noting Apple does have its own enterprise identity (Managed Apple IDs), but that's separate. Entra ID integration on iOS is purely in the context of apps and web authentication, not the device's Apple ID. This means the user will still unlock the device with a personal PIN/biometric, but as soon as they try to access corporate resources, the Entra ID SSO kicks in. Apple's iOS also supports Account-driven User Enrollment (where an MDM enrollment using a Managed Apple ID can create a separate work partition). In Microsoft's ecosystem, that's more used for BYOD with a lighter management. Even in those cases, the Microsoft SSO plug-in can function within the managed account context. So Intune supports both full device enrollments and user enrollments with SSO, though the seamlessness is greatest with full enrollment. 

Overall, on iOS, Intune + Entra ID provide app-level SSO and authentication brokerage. A user with an Intune-managed iPhone might still authenticate once (enter credentials in Authenticator or Company Portal during setup), but afterwards, they often won't have to enter their password again on that device - not for Outlook, not for Teams, not even for viewing an intranet site in Safari, thanks to the SSO plug-in75 76. And because of device registration, that authentication is strong (could be device-bound passkey in Authenticator) and tied to a compliant device, satisfying high security requirements without user friction. 

# [:::image type="icon" source="../../media/icons/platforms/android.svg"::: **Android**](#tab/android)

Android's scenario is analogous to iOS in many ways, with a few platform-specific differences: 

Device Enrollment and Work Profile: Intune can enroll Android devices in various modes (Work Profile for BYOD, Fully Managed, Corporate-Owned with Work Profile, etc.). In each case, installing the Intune Company Portal app is usually the first step. During enrollment, the user is directed to sign in with their Entra credentials, and the device becomes registered in Entra ID (just like iOS) and known to Intune. Depending on the scenario, either the Company Portal or the Authenticator app will act as the authentication broker on Android. On Android, both apps have broker capabilities77. Typically: 

If only Company Portal is present (and user hasn't installed Authenticator yet), Company Portal will handle token requests in the interim. 

Microsoft's recommendation these days is often to use Authenticator as the primary broker on Android as well, so many organizations push Authenticator via Intune and have users sign in there. 

If using a Work Profile (the standard BYOD mode), both Company Portal and Authenticator will be installed inside the work profile. The "broker" tokens (PRT) are stored within that work profile, isolated from personal side. 

SSO and Broker Functionality: Once enrolled, the experience is similar to iOS: the first app that needs sign-in (maybe Outlook) will cause a redirect to the broker (Authenticator or CP). The user might do a full authentication (possibly including MFA). Entra ID then issues a Primary Refresh Token that is stored by the broker app on Android78. After that, any app that is MSAL-enabled (Microsoft's auth library) will check for a broker token. All Office apps, Teams, etc., will find the token and silently log the user in. Even third-party apps using Entra ID can leverage the broker if they are configured to. Unlike iOS, Android did not have until recently a native SSO extension framework. However, Android 11+ introduced a feature called "Cross-profile SSO" for work profiles, which Microsoft supports so that once you sign in to one app in the work profile, others can use that identity. Intune facilitates that by provisioning the necessary dependencies and permissions during enrollment. 

Passwordless and Passkeys on Android: Users can also go passwordless with Authenticator on Android. The phone sign-in (Auth app showing "Approve/Deny" or code matching) works similarly to iOS. Additionally, with Android 14, Authenticator can store an Entra ID passkey (device-bound) just like on iOS79 80. The user would enable the phone's screen lock and biometric, update Authenticator, and add a passkey for their account. Intune ensures the device meets requirements (Intune compliance might require a screen lock PIN of a certain complexity on Android anyway, which is prerequisite for using any passkey). For corporate scenarios on Android, especially when using a Work Profile, it's important that the passkey is added in the Work Profile's Authenticator instance (since that's where corporate apps will look). Guidance in Microsoft docs to users (and IT) is to ensure to add the passkey in the managed profile Authenticator, not just personal side81 82. Intune can help by delivering user instructions via the Company Portal branding or notifications, but mostly this is a user education step. 

Conditional Access and Compliance: Similar to iOS, Conditional Access can treat an Intune-managed Android as a compliant, valid device. A policy might require Android devices be marked compliant and use an approved app. Intune's compliance policies for Android (encryption, OS version, no rooted device, etc.) feed into that. If a user tries to sign in from an unmanaged Android, CA can block it or require additional factors. If managed but not compliant, they get blocked until fixed. The integration ensures that even if the user has a broker token, Entra ID will check device state on each token usage for apps with certain sensitivity. 

Differences in UI and User Experience: On Android, the Authenticator app also recently gained the ability to be set as the device's default password manager & passkey provider (akin to iOS's feature)83 84. If a user does that, it can autofill credentials and also handle passkeys at the OS level. This might slightly blur lines between personal and work usage of passkeys, but Intune's focus would be ensuring corporate accounts are protected. Also, on Android, because there are many device manufacturers, some slight differences exist (e.g., Samsung devices with Knox might require additional steps). However, from Intune/Entra's perspective, the flow is standard. 

In summary, on Android devices Intune establishes a managed context (which Entra ID recognizes), and Entra ID via Authenticator provides the single sign-on backbone. The result: a user enrolls their Android phone, signs in once, and then they can use all their authorized apps without repeated sign-ins. They might even use biometric unlock for the Authenticator app to satisfy MFA prompts instead of passwords. The combination of Intune and Entra ID ensures that is both seamless and secure: seamless, because the token sharing is automatic; secure, because the device is managed and must remain compliant for those tokens to be valid.

## Intune Policies and Configuration Options Across Platforms 

Bringing together the above, it's useful to highlight some of the key Intune configuration options that administrators use to enforce and optimize passwordless sign-in on various platforms: 

# [:::image type="icon" source="../../media/icons/platforms/windows.svg"::: **Windows**](#tab/windows)

In Intune, admins use Windows Enrollment > Windows Hello for Business to turn on Hello and optionally set it tenant-wide. They also use Configuration Profiles (Identity Protection template) to fine-tune Hello or enable FIDO2 sign-in. Additionally, the Settings Catalog now includes the Authentication category for Windows 10/11, where settings like Hide Password credential (Enable passwordless experience) and Enable Web Sign-In are found85 86. Typically, an admin creates a Settings Catalog profile setting Enable Web Sign-In = Enabled (to support TAP and phone sign-in at logon) and Enable Passwordless Experience = Enabled (to hide the password box)87 88, and deploys that to Windows 11 devices. These two settings complement the Hello for Business policy. Intune also can set default credential provider order via OMA-URI if needed (so that, for example, Hello is the default over Web Sign-in at the login screen, as a blog recommended89 90). 

# [:::image type="icon" source="../../media/icons/platforms/macos.svg"::: **macOS**](#tab/macos)

Intune's Settings Catalog for macOS includes the Extensible Single Sign-On payload. Within that, an admin can configure Platform SSO by filling in fields like the extension identifier (for Microsoft's plug-in), Azure AD tenant info, and choosing the auth method (UserSecureEnclaveKey for passwordless, or Username/Password, etc.)91 92. This is a one-stop configuration that Intune pushes as a device profile. Intune documentation and portal make it clear that if you use Platform SSO, you shouldn't also use a separate SSO extension profile - the single profile suffices93 94. Also, Intune can deploy the Company Portal app required for this via the macOS client apps (Intune has it in the Mac App Store). Other settings Intune can manage: enabling FileVault (disk encryption) which might be a compliance requirement, and mapping Active Directory Kerberos extension if needed (for on-prem SSO if the org has a local AD, though that's separate from Entra ID SSO). Another relevant setting is "Require users to log in to the device with Azure AD/Microsoft Entra ID" which Intune now supports for Platform SSO - essentially forcing the use of Entra ID login on the Mac once configured, rather than letting users opt out. 

# [:::image type="icon" source="../../media/icons/platforms/ios-ipados.svg"::: **iOS/iPadOS**](#tab/ios-ipados)

For iOS, Intune administrators use the Device Features > Single sign-on app extension template (or Settings Catalog equivalent) to set up the Microsoft Azure AD SSO extension95 96. They will specify the Type as "Redirect" and choose Microsoft Entra ID as the extension, and provide required fields (like the tenant ID, and a list of URL prefixes that should leverage SSO - for example, https://login.microsoftonline.com, and any specific app URL schemes). The Intune docs provide guidance on these values so that once deployed, any app that uses modern auth will invoke this extension97 98. Intune also can push the Authenticator app to iOS devices (either as a required app or availability in the Company Portal for users to install). Many orgs make Authenticator a required install on managed iOS as part of baseline security. Compliance policies on iOS often require a device PIN of certain length/complexity - indirectly this ensures that if Authenticator is used for passkeys, the device has a solid security baseline (since iOS uses the device PIN/biometric to unlock the credential store). 

# [:::image type="icon" source="../../media/icons/platforms/android.svg"::: **Android**](#tab/android)

Intune's approach for Android is to require certain device conditions (like a PIN, encryption, no developer mode) via compliance policy. Administrators also often use Conditional Access rules requiring "approved client app" or "App Protection Policy" which in effect means the user must use apps like Outlook or Teams (which are broker-aware) rather than random apps. Intune can deploy Authenticator on Android work profile automatically during enrollment (by marking it as required for the managed Google Play). If the organization uses Android Enterprise Corporate-Owned Work Profile (COPE) or Fully Managed, Intune configures the Managed device to have the necessary accounts and can even enable Shared device mode for Authenticator if the device is shared among users. The key Intune setting specifically for SSO is enabling the broker and ensuring the device is Azure AD registered. There is not an "SSO extension" on Android to configure (as of now), because the broker handles it. However, Intune does have some settings for Android Enterprise that can impact auth — for example, with Android 12+, Google introduced a feature where the Company Portal can pass the device registration to the account manager without needing the user to go through a separate Company Portal login flow (this is more internal detail; from admin side, not much to set beyond ensuring the latest CP and Authenticator). 

---

Conditional Access "Authentication Strength": While not an Intune setting, it's worth noting as a policy option in Entra ID that pairs with Intune-managed credentials: admins can create Conditional Access policies that require specific auth strengths (like "Require phishing-resistant MFA"). This strength can be satisfied by Hello for Business, FIDO2, or certificate-based auth - but not by a mere password or SMS code. By using this in CA, organizations effectively force users to use passwordless methods (because a password alone or a weak MFA won't allow access)99 100. Intune's role is to have set up those methods on devices so that users can meet the requirement. For example, if "phishing-resistant MFA" is required, the user must either sign in with Hello (which counts as two-factor by itself) or do a FIDO2 key or Authenticator passkey - all of which Intune has prepped them for. If they try a password + SMS, it would be blocked. This tandem of Intune device configuration and Entra ID policy ensures a passwordless mandate is technically enforced. 

All these policies come together to allow IT to sculpt how users authenticate on each device type. A typical strategy could be: enable Windows Hello and hide passwords on Windows devices; enable Platform SSO on all Macs; push Authenticator with SSO extension on mobile; require MFA or passwordless authentication via Conditional Access. Intune provides the device-side, and Entra ID provides the gatekeeping side.
